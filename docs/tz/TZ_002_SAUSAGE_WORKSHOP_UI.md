# TZ 002: Sausage Workshop UI

## Objective

Создать техническое задание на отдельное frontend-приложение
`sausage-workshop` для управления колбасным цехом.

Приложение должно быть отдельным рабочим пространством, а не вкладкой внутри
Siyoma dashboard. Визуальный и продуктовый референс:

```text
/Users/hafizov/Downloads/sausage_workshop_dashboard.html
```

## Context

Siyoma остается host platform на стартовом этапе, но колбасный цех является
отдельным доменом `sausage-production`.

`sausage-workshop` должен использовать API namespace:

```text
/api/sausage-production/*
```

Нельзя использовать `/api/production/*` для нового frontend.

## Goals

- Зафиксировать структуру отдельного цехового интерфейса.
- Перенести из HTML-референса рабочую модель: sidebar, topbar, KPI, таблицы,
  быстрые операции, журналы, партии, потери.
- Подготовить frontend так, чтобы сначала он мог работать на mock data, а
  затем переключиться на real API.
- Сохранить возможность вынести frontend в самостоятельный deployment.

## Non-Goals

- Не писать backend.
- Не создавать Prisma schema.
- Не копировать backend Siyoma.
- Не встраивать этот интерфейс как основную вкладку в Siyoma dashboard.
- Не делать мобильное приложение цеха.
- Не делать полный HACCP, этикетки, QR, серийную маркировку в этом ТЗ.

## Core UX Principle

Интерфейс должен быть рабочей панелью для менеджера/мастера цеха.

Это не landing page и не маркетинговый экран. Первый экран должен сразу
показывать:

- состояние сырья;
- состояние цеха;
- готовую продукцию;
- потери;
- активные производственные заказы;
- критические остатки;
- последние события;
- быстрые действия.

## Visual Direction

Визуальный стиль по референсу:

- темная рабочая поверхность;
- плотная layout-сетка;
- левый sidebar;
- верхняя панель действий;
- моноширинные номера документов и числовые значения;
- акцентный цвет для действий и важных объектов;
- таблицы как основной носитель данных;
- компактные KPI;
- четкие статусы и progress bars;
- минимум декоративных элементов.

Избегать:

- hero sections;
- marketing copy;
- крупных декоративных карточек;
- градиентных фонов;
- интерфейса, похожего на общий SaaS landing.

## Data Strategy

В Phase 1 frontend может использовать mock data, но структура mock data должна
совпадать с будущими API DTO.

Mock data не должен быть размазан по компонентам. Нужен отдельный слой:

```text
mockSausageProductionApi
```

Позже он заменяется на:

```text
sausageProductionApiClient
```

## API Client Requirement

Frontend должен обращаться только через client abstraction.

Запрещено:

```text
fetch("/api/sausage-production/...") внутри UI-компонентов
```

Нужно:

```text
sausageProductionApi.listOrders()
sausageProductionApi.createRawReceipt(input)
sausageProductionApi.completeProductionOrder(id, input)
```

## Responsiveness

Desktop-first.

Минимальные требования:

- 1440px: полноценный sidebar + таблицы;
- 1200px: таблицы остаются читаемыми;
- 768px: sidebar может скрываться;
- mobile не является главным сценарием ТЗ-002.

## Accessibility

Минимальные требования:

- все actions доступны через button/link, не только div;
- focus state видим;
- статус не передается только цветом;
- таблицы имеют читаемые заголовки;
- modal можно закрыть кнопкой.

## Implementation Constraints

Когда начнется реализация:

- не копировать backend Siyoma;
- не использовать `/api/production`;
- не называть приложение `production-dashboard`;
- не смешивать UI с Siyoma dashboard;
- не создавать shared state, завязанный на beverage/water domain;
- не добавлять прямые imports из Siyoma frontend.

## Acceptance Criteria

- `sausage-workshop` существует как отдельное frontend-приложение.
- Первый экран визуально и структурно следует HTML-референсу.
- В sidebar есть все разделы из ТЗ.
- Dashboard показывает KPI, quick actions, активные заказы, критические
  остатки, события, выпуск за день и готовую продукцию.
- Есть отдельные экраны заказов, партий, перемещений, сырья, готовой продукции,
  рецептур, потерь и аналитики.
- UI использует mock API/client abstraction, а не hardcoded данные в каждом
  компоненте.
- В коде и routes используется `sausage` namespace.
- Нет интеграции с Siyoma dashboard как основной реализации.
- Нет API calls на `/api/production/*`.
